:::tip 简洁的框架
## **简洁**： 集合美妙优点　
  DopTime 最核心想法是尽可能取消代码。对人更友好, 为AI代码生成高度优化
:::
- **简洁**：仅使用Redis操作数据。高并发，快响应。同时代码简洁。
- **简洁**：当你定义了一个函数，它就是一个可以被RPC调用的微服务。也同时可以通过http调用。
- **简洁**：当你定义了一个函数的入参和出参。就可以替代使用生成器， 免于一切学习和繁琐的数据多次定义。
- **简洁**：数据的增删查改基本靠REDIS语句，你都不需要写CURD。
- **简洁**：不再区分单体服务和微服务。你的代码，单体服务, 也是微服务。但你想从单体切换到多体时，它已经是了。
- **简洁**：你随时可以像把网线插拔到交换机一下，热插拔你的软件。
- **简洁**：你只需专注核心业务代码，用心流的方式  
  &nbsp;
:::tip  性能
## **性能**：最大化并发与最小化延迟
:::

- **性能**：dragonflydb, 可以提供每秒2M+的读写性能。  
- **性能**：对本地数据的读写延迟对应redis缓存的延迟。它的性能大大高于mysql。基本都在1ms以内。

:::tip 成本
## **成本**：如何砍掉大部分
:::

- **成本压缩**：砍掉重复开发  
删掉重复开发。许多公司都提供了API接口。但你还需要把API用起来。这是重复繁琐的工作。  
如果你封装了一个接口，你应该把它发布到doptime的公开平台上。这样其它人可以直接调用你的接口。避免重复的开发。
doptime的目标是把这个重复性工作压缩到几乎没有。  


- **成本压缩**：使用最低价api  
随着AI的发展，AI 服务的引入不可避免。越来越昂贵的ai硬件使得AI的服务又注定会以API的方式流行。doptime 倾向于认为我们需要建立一个基于大宗交易的竞价模式，把所有必须得服务费用降到最低。doptime 欢迎你发布这样的api.

- **摆脱API**：
doptime倾向于认为你不应该部署自己的API。  
AI的发展，意味着API的实现成本在指数级降低。它的价值是快速贬值的，Doptime 也致力于这种贬值。
API的部署应该需要非常专业化的方案来实现。你应该部署自己的数据服务就够了。
因为有价值的是数据，而不是达成操作这些数据的动作。

- **成本压缩**：把服务部署在本地，而非云硬件上。  
云硬件的毛利通常很高。更合理的，我们应该把服务部署在自己的本机上。这通常要便宜很多，而且本地通常有数以TB计的硬件资源，和大量的cpu 核心。
同时，我们需要确保以符合区域法规的方式做到这一点。
